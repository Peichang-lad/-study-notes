## 前言

如果你接触过 Linux 文件系统，一定听说过 inode（index node）。简单来说，inode 是文件系统中用来描述文件元数据的结构体，包含文件大小、权限、所有者、时间戳、数据块指针等信息。但有一个问题经常让人困惑：**inode 里保存了这么多信息，为什么偏偏不保存文件名？**

## inode 的职责

inode 的本质是"文件的身份证"。一个文件在磁盘上对应一个唯一的 inode，操作系统通过 inode 来定位文件的数据块。inode 中的关键字段包括：

- **文件类型和权限**：是普通文件、目录还是符号链接？读、写、执行权限如何？
- **文件大小**：文件占用了多少字节？
- **链接计数**：有多少个文件名指向这个 inode？
- **时间戳**：创建时间、最后修改时间、最后访问时间。
- **数据块指针**：指向实际数据存放的磁盘块。

这些信息都是文件本身固有的属性，与文件叫什么名字没有本质关系。

## 文件名属于目录

文件名实际上存储在**目录项（directory entry）**中，而不是 inode 中。目录本质上也是一个文件，它的数据块里保存的是一组 `(文件名 → inode 号)` 的映射表。当我们执行 `ls` 或 `cat hello.txt` 时，操作系统做的事情是：

1. 在当前目录的数据块中查找 `hello.txt` 对应的 inode 号；
2. 通过 inode 号找到对应的 inode 结构体；
3. 从 inode 中读取数据块指针，最终拿到文件内容。

## 为什么这样设计？

这种设计带来了一个非常重要的特性：**硬链接（hard link）**。

如果文件名保存在 inode 里，那么 inode 与文件名就是一一对应的，一个文件只能有一个名字。但实际上，inode 和文件名是多对多的关系——同一个 inode 可以有多个文件名指向它（硬链接），只要 inode 中的链接计数大于 0，文件就仍然存在。

```bash
# 创建一个硬链接
ln original.txt link.txt

# 两个文件名指向同一个 inode
ls -li
# 输出示例：
# 123456 -rw-r--r-- 2 user group 1024 Jul 8 10:00 original.txt
# 123456 -rw-r--r-- 2 user group 1024 Jul 8 10:00 link.txt
```

可以看到，两个文件的 inode 号完全相同（123456），链接计数都是 2。删除 `original.txt` 只是将链接计数减 1，文件数据本身只有在链接计数归零时才会被真正释放。

## 软链接 vs 硬链接

与硬链接不同，**符号链接（软链接）**有自己的 inode，它的数据块里保存的是目标文件的路径字符串。这就是为什么软链接可以跨越文件系统，而硬链接不可以——inode 号只在同一个文件系统内唯一。

```bash
ln -s original.txt soft.txt
ls -li
# soft.txt 有自己独立的 inode，类型是 l（符号链接）
```

## 总结

| 结构 | 存放位置 | 存储内容 |
|------|----------|----------|
| inode | inode 表 | 权限、大小、数据块指针、链接计数 |
| 目录项 | 目录数据块 | 文件名 → inode 号 |

不把文件名放入 inode 是一层巧妙的**解耦设计**：它使得文件名只是文件的一个标签，多个标签可以贴到同一个文件上，而文件的数据和元数据完全不受标签变化的影响。这种设计也为后来的符号链接、挂载点等概念奠定了基础。理解这一点，是理解 Linux 文件系统架构的关键一步。
