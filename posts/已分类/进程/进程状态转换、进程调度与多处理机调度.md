# 进程状态转换、进程调度与多处理机调度

## 一、进程状态转换

### 基本三态模型

```
        创建
          ↓
       就绪态 (Ready)
      ↗        ↘
 运行→就绪      就绪←阻塞
      ↘        ↗
       运行态 (Running)
          ↓
       阻塞态 (Blocked)
```

更清晰的状态转换图：

```
新建(New) ──创建完成──→ 就绪(Ready)
                            ↑  ↓ 调度程序选中
                    时间片到 │  ↓
                         运行(Running) ──完成──→ 终止(Exit)
                            │
                       等待事件│
                            ↓
                        阻塞(Blocked)
                            │
                       事件发生│
                            ↓
                         就绪(Ready)
```

### 五态模型（加入挂起）

|状态|说明|
|---|---|
|新建 (New)|进程刚被创建，尚未进入就绪队列|
|就绪 (Ready)|具备运行条件，等待 CPU|
|运行 (Running)|占用 CPU 正在执行|
|阻塞 (Blocked)|等待某事件（I/O、信号量等）|
|挂起 (Suspended)|被换出内存，分为就绪挂起 / 阻塞挂起|

### 关键转换触发条件

|转换|触发原因|
|---|---|
|就绪 → 运行|调度程序选中该进程|
|运行 → 就绪|时间片耗尽、高优先级抢占|
|运行 → 阻塞|主动等待 I/O、请求资源、P操作阻塞|
|阻塞 → 就绪|等待的事件完成（I/O结束、V操作唤醒）|
|运行 → 终止|进程执行完毕或被强制终止|

> ⚠️ **注意**：阻塞不能直接→运行，必须先回到就绪态再等待调度。

## 二、进程调度

### 调度层次

```
长程调度（作业调度）
   ↓  决定哪些作业进内存
中程调度（内存调度）
   ↓  决定进程换入/换出
短程调度（CPU调度）
   ↓  决定哪个就绪进程获得CPU
```

### 主要调度算法

**① 先来先服务 FCFS**

- 按到达顺序排队，非抢占
- 优点：公平，实现简单
- 缺点：平均等待时间长，对短进程不友好（护航效应）

**② 短作业优先 SJF**

- 选运行时间最短的进程
- 非抢占版：等当前进程结束再调度
- 抢占版（SRTF）：新进程剩余时间更短则立即抢占
- 优点：平均等待时间最优
- 缺点：长进程可能饥饿，需预知运行时间

**③ 优先级调度**

- 每个进程有优先级，选最高优先级执行
- 静态优先级 vs 动态优先级（老化机制防止饥饿）
- 可抢占或不可抢占

**④ 时间片轮转 RR（Round Robin）**

- 每个进程分配固定时间片，轮流执行
- 时间片大小选择关键：太小→上下文切换开销大；太大→退化为FCFS
- 适合分时系统，响应时间好

**⑤ 多级反馈队列 MLFQ**

- 多个优先级队列，高优先级时间片短
- 新进程进入最高优先级队列
- 用完时间片未完成则降级；等待过久则升级（防饥饿）
- 综合了 RR 和优先级的优点，是最通用的算法

### 调度性能指标

|指标|说明|
|---|---|
|CPU利用率|CPU有效工作时间占比|
|吞吐量|单位时间完成的进程数|
|周转时间|完成时刻 - 提交时刻|
|等待时间|在就绪队列中等待的总时间|
|响应时间|提交请求到首次响应的时间（交互系统关键）|

## 三、多处理机调度

### 核心概念区分

|类型|说明|
|---|---|
|SMP（对称多处理）|所有CPU地位平等，共享内存，最常见|
|AMP（非对称多处理）|主CPU负责调度，从CPU执行任务|
|NUMA|非一致性内存访问，访问本地内存快，远程慢|

### 关键设计问题

**1. 就绪队列结构**

```
方案A：全局单一队列          方案B：每CPU私有队列
┌──────────────┐            CPU0队列  CPU1队列
│ 全局就绪队列  │            [P1][P3]  [P2][P4]
└──────┬───────┘
    ┌──┴──┐
   CPU0  CPU1
```

- 全局队列：负载均衡好，但锁竞争是瓶颈
- 私有队列：无锁竞争，但可能负载不均，需要**工作窃取（Work Stealing）**

**2. 处理机亲和性（Affinity）**

进程倾向于在同一个 CPU 上运行，原因：该 CPU 的 Cache 中有该进程的热数据，迁移会导致 Cache Miss

- **软亲和性**：尽量不迁移，但允许
- **硬亲和性**：强制绑定某个 CPU（如实时任务）

**3. 负载均衡**

- **推（Push）迁移**：定期检查各CPU负载，将任务从繁忙CPU推走
- **拉（Pull）迁移**：空闲CPU主动从繁忙CPU拉取任务
- 负载均衡与 Cache 亲和性存在**天然矛盾**，需要权衡

**4. NUMA 感知调度**

```
Node 0              Node 1
CPU0 CPU1          CPU2 CPU3
  └──┬──┘            └──┬──┘
  内存0  ←─慢速互联─→  内存1
```

调度原则：优先将进程调度到其内存所在节点的 CPU，减少跨节点访问延迟

### 多处理机特有调度策略

|策略|说明|适用场景|
|---|---|---|
|协同调度（Gang Scheduling）|线程组同时上多个CPU运行|紧密通信的并行程序|
|专用处理机分配|每个进程固定一个CPU|高性能/实时场景|
|动态调度|进程自己决定何时让出CPU|支持用户态调度的系统|

## 总结串联

```
进程创建
   ↓
进入就绪队列 ←─────────────────────────────┐
   ↓                                        │
调度算法选择（FCFS/SJF/RR/MLFQ）           │
   ↓                                        │
分配CPU（单处理机/多处理机）                │
   ↓                                        │
运行 ──时间片到/抢占──→ 就绪队列（循环）   │
   │                                        │
   ├──等待I/O──→ 阻塞 ──I/O完成──→ 就绪 ──┘
   │
   └──完成──→ 终止
```

多处理机在此基础上额外考虑：**哪个CPU执行**（亲和性）+ **各CPU是否均衡**（负载均衡）+ **内存距离**（NUMA感知）
