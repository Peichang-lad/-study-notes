# 管程（Monitor）

---

## 为什么需要管程？

信号量机制的**缺点：**

- PV操作分散在各个进程中，容易**写错顺序**
- 错误难以发现和调试
- 编程复杂，负担重

管程的目标：**把分散的同步操作集中起来，封装成一个模块**，让程序员更安全地写并发代码。

---

## 管程是什么？

> 管程 = **共享资源** + **操作这些资源的一组过程（函数）** + **互斥保证**

类比理解：**管程就像一个加锁的房间**

- 房间里有共享数据
- 只有通过门（过程/函数）才能进入
- 同一时刻只有一个人能进入（互斥自动保证）

---

## 管程的三个核心特性

|特性|说明|
|---|---|
|**互斥性**|任意时刻只有一个进程在管程内执行，自动保证，无需手动加锁|
|**封装性**|共享变量只能通过管程提供的过程访问，外部不能直接读写|
|**条件变量**|用于在管程内部实现同步（进程等待/唤醒）|

---

## 条件变量（Condition Variable）

这是管程最重要的概念，用来解决**"资源不满足时如何等待"**的问题。

每个条件变量对应一个**等待队列**，只有两个操作：

```
x.wait()   → 当前进程挂起，释放管程，进入x的等待队列
x.signal() → 唤醒x等待队列中的一个进程
```

**与信号量的区别：**

- 信号量 V操作：即使没人等，值也会+1（留存）
- `x.signal()`：若无人等待，**什么都不发生**（不留存）

---

## 用管程解决生产者-消费者

```
Monitor ProducerConsumer:
    int count = 0         // 缓冲区产品数
    condition full        // 缓冲区满时生产者在此等
    condition empty       // 缓冲区空时消费者在此等

    procedure put():      // 生产者调用
        if count == N:
            full.wait()   // 满了就等
        放入产品
        count++
        empty.signal()    // 唤醒消费者

    procedure get():      // 消费者调用
        if count == 0:
            empty.wait()  // 空了就等
        取出产品
        count--
        full.signal()     // 唤醒生产者
```

**对比信号量版本：** 不需要手动写P(mutex)，管程自动互斥，干净很多。

---

## 唤醒后谁先执行？两种策略

执行 `x.signal()` 唤醒等待进程后，存在冲突：**唤醒者和被唤醒者都想进管程**

|策略|说明|
|---|---|
|**HSU（Hansen）**|唤醒者继续执行，被唤醒者等唤醒者退出后再进|
|**Hoare**|唤醒者立刻让出管程，被唤醒者先执行|

> 考研一般不深究哪种策略，了解概念即可。

---

## 管程 vs 信号量 对比

|对比项|信号量|管程|
|---|---|---|
|互斥实现|手动P/V|自动保证|
|编程难度|高，容易出错|低，更安全|
|同步机制|信号量值|条件变量|
|使用场景|底层OS实现|高级语言并发（Java的synchronized）|

---

## 一句话记忆

> **管程 = 自动互斥的共享资源管理模块，用条件变量替代信号量实现同步，是信号量的高级封装。**

Java中的 `synchronized` 关键字本质上就是管程的实现！
